Mercury：一种用于加速分布式训练的传输层调度器

发表于 2022-05-24 分类于 论文阅读 阅读次数： 阅读次数： Valine：

并行训练的大规模深度学习模型的效率依赖分布式集群间的通信调度，优化时间开销的常规方法包括：Tensor Fusion 和 Tensor Partition。后者能够以更细的粒度实现优先级调度，但这需要在每个张量分区之前插入一个启动时隙，反而会抵消其带来的调度增益。此外，调整最佳分区大小也很耗时，所以现有的应用层解决方案无法消除分区开销。

基于此背景，文章提出了 Mercury，一个简单的传输层调度器，在通信和计算之间实现了近乎最佳的重叠。它不对张量进行分区，而是将优先级调度以数据包粒度移动到传输层，缓冲区中优先级最高的数据包将首先被传输。 它利用传输层的即时聚合来实现一致的梯度推送和参数拉取。

从实验结果来看，Mercury 在带宽资源受限的环境和通信时间开销占主导的场景下能够比以往优化方法取得更加卓越的效果。

并行数据的分布式深度学习将数据划分为多个分片，每个分片由单独的工作节点放置和计算。一种广泛使用的架构为参数服务器（PS）架构，节点在逻辑上被归类为“服务器”，需要保持模型参数的全局统一性，以及执行本地 DNN 训练的 “Worker”

通信瓶颈：

模型愈来愈大，服务器集群通信带宽却有限。花在通信上的时间比本地计算梯度的时间要大得多，成为整体模型训练的瓶颈。

计算、通信的执行策略

很自然地，反向传播结束后才能更新参数，然后 worker 才能拉取最新模型参数，进而才能推送本地计算的梯度值。前向传播、反向传播消耗计算资源，而 push 使用上行带宽、 pull 使用下行带宽，他们消耗的资源是不同的。因此如果计算时间与通信的执行能够一定程度上并行进行，则可以有效减少训练时间消耗。

下面以一个3层神经网络的 tensor 调度时间使用作为例进行对比，文章做如下假设：

  1. 设某一层的参数用一个 tensor 表示
  2. 所有 worker 推送、拉取一个 tensor 的操作时间完全一致
  3. PS 可以瞬间完成 worker 传来的梯度值的累加
  4. 不同层参数 tensor 大小不同，设 tensor2 > tensor3 > tensor1

WFBP策略

WFBP 是目前主流深度学习框架采用的默认策略，梯度值在完成反向传播后才会调度传输。如图所示，bp3 完成后 worker 才 push 给 PS，push 操作消耗的时间包括：启动时间、传输时间。TS1 推送完后 worker 才会拉取TS3 对应的更新模型参数并推送 TS2 的梯度值。很明显，WFBP 并没有并行进行计算和通信操作。

用户级Tensor分区策略

Tensor 分区和优先级调度能够改善计算-通信并行性。如下图所示，在分布式 DNN 训练中，每一层都可以细分为多个较小的张量进行细粒度的优先级调度。具有更小标号的 tensor 拥有更高优先级，可以看见两次bp3的 push 后，进行了一次bp2的 push，而由于此时bp1完成了计算，因此暂停bp3、bp2的push，优先推送bp1。这种抢占式调度与 Tensor 分区一起提前了 fp1 的执行，同时调度器可以利用 Tensor 分区来更好地并行化推拉操作。

但是在实际传输 tensor 时，需要先进行一系列的额外处理，包括：RPC序列化、ZeroMQ消息排队和处理器核心调度，其消耗的总时间称为启动时间，并与 tensor 大小无关。所以，Tensor 分区带来更多的传输次数，同时也会导致更多的启动时间消耗。

启动时间带来的问题

  1. 带宽利用率不足

    由于 Tensor 分区，每次传输时间变短，相较之下启动时间开销增大，期间信道带宽就被浪费了

  2. 分区策略参数调整的低效

    用户级 tensor 分区需要调整 tensor 的大小和调度抢占。虽然最小的 tensor 大小能带来最佳的优先级调度灵活性，但是太细粒度的 Tensor 划分会导致每轮训练的总启动时间更长。tensor 大小对训练速度的影响如图所示，这是一台 PS 和一台 worker 在不同通信带宽下不同 tensor 设置的训练速度对比：

    有人提出过一种解决方案：基于信用值的抢占策略

    信用值定义为允许填充在网络堆栈发送缓冲区中 tensor 的最大容量。这些缓冲 tensor 可以在底层 FIFO 队列中同时传输。由于更早的张量的传输时间与后续张量的启动时间重叠，网络利用率得到了提高。不过这样会导致基于优先级的调度受到破坏，因为传递到底层队列的张量不能再按优先级来抢占。下图说明了训练速度如何受到不同带宽设置和不同 DNN 模型的信用值的影响：

    选择最佳分区大小和信用值是个很复杂的工作，因为它们会受到带宽、计算硬件、DNN 框架和 DNN 模型的影响，并且它们的相互依赖关系也很难建模。

    尽管 ByteScheduler 使用贝叶斯优化方法来自动调整系统参数，但其搜索时间长达几分钟，无法适用于动态网络环境，特别是当带宽由其他通信任务共享时。

综上，文章认为在存在启动时间的情况下，上层解决方案很难实现最佳的通信和计算的并行处理，因此提出了一种在传输层的解决方案：Mercucy。

Mercucy 系统设计

核心概念

从用户级张量划分的问题可知，在 DNN 引擎之上部署中间件是很难实现完美的计算-通信并行的。文章提出在不牺牲抢占机会的情况下将基于优先级的调度下移到传输层。

  1. 启动时间最小化

    一旦 BP 操作完成，梯度 tensor 就直接传递到底层传输队列。不存在张量分区，也不会引入额外的启动时隙

  2. 传输层调度

    通过配置多传输队列来实现基于优先级的调度。第 i 个张量的数据包存放在第 i 个优先级队列中，优先级最高的总是最先传输，相当于数据包级别的优先级调度和抢占

  3. 即使 PS 聚合

    PS 在数据包到达时聚合数据包中的梯度，并在所有 worker 梯度的聚合完成后将它们发送回数据包。也就是说拉取(pull)操作是数据包粒度的

Mercury 的时间使用如图所示，推操作只有三个启动时隙，并且推拉操作几乎完全重叠。与 WFBP 和 Tensor分区 相比，Mercury 大大减少了时间开销。

启动时间根本上来自分布式 DNN 引擎的工作流程。自顶向下，通信栈包括：用户代码、引擎核心、基于消息的通信库和 TCP 或 RDMA 传输。用户代码将推送和拉取操作及其关联的 tensor 封装到运算符中。引擎核心根据它们的依赖关系确定通信和计算算子的执行顺序。基于消息的通信库通常由 RPC、ZeroMQ、MPI 和 NVIDIA NCCL 组成。当一个大 tensor 的 push 操作在用户代码中被划分为多个算子时，它们必须被多次传递，从而导致更多的启动延迟。Mercucy避免了过多的启动间隔，因为每个 tensor 都被传递到通信堆栈而没有分区。

基于优先级的 tensor 调度也是 Mercury 的关键部分。但是，传输层的传输队列是无视内容的。文章的提议只是将这个 FIFO 传输队列转换为 tensor 优先级队列。与用户级 Tensor 划分相比，优先级调度和抢占是在数据包级实现的。默认情况下，每一层的 tensor 都有其唯一的优先级，当层数过多时，多个 tensor 可以共享相同的优先级。即时 PS 聚合保证了在没有张量分区的情况下推拉操作的重叠。否则，Mercury 就退化为 WFBP。

系统概述

Mercury 由两个组件组成：Mercury Worker 和 Mercury PS，下图显示了训练期间工人和 PS 之间的交互。用户级 Worker 和 PS 是由 DNN 引擎实现的高层抽象概念。 Worker 迭代执行 BP 和 FP，而 PS 聚合全局模型参数。 Mercury Worker 和 Mercury PS 充当传输库，将梯度 tensor 从 worker 传到 PS，参数 tensor 从 PS 传到 worker。

Mercury Worker 执行基于数据包级别优先级的调度，且不会降低吞吐量。当 BP 产生的梯度 tensor 传递给 Mercury Worker 时，tensor 被细分为 Mercury 数据包。 Mercury Worker 根据 tensor 的层号为每个 Mercury 数据包分配一个优先级。优先级最高的数据包总是最先发送。 Mercury Worker 还从 Mercury PS 接收参数数据包，合并形成参数 tensor 后将其馈送到 FP。

Mercury PS 执行即时数据包级梯度聚合。在批量同步并行 (BSP) 模式下，聚合仅包含简单的计算（比如对所有 worker 的梯度取平均），几乎不导致 CPU 开销。因此 Mercurcy 将聚合操作移动到传输层以方便立即拉取。

Mercury PS 使用代理模块来聚合梯度并更新模型参数，无需将 tensor 传递给 PS 用户代码。这样做有效地减少了 tensor 在数据包粒度上的处理延迟。否则，每个数据包都必须从最低的传输层传到最高的应用层进行聚合，然后返回传输层进行传输，由于通信堆栈处的非平凡延迟，导致聚合速度减慢。

实现细节

Mercucy Worker

Sending Buffer：包含多个 Mercury 数据包队列

Worker Sender：从发送缓冲区中逐一提取数据包

Transport Library：实际进行传输操作

Tensor Constructor：接收 Mercucy Packets 并构造参数 tensor

Sending Buffer 中的队列数等于调度优先级的总数。通常数据包的数量远大于调度优先级的数量。为了降低提取数据包的时间复杂度，Mercucy 使用一组 FIFO 队列而不是单个堆优先级队列。每个 FIFO 队列都唯一对应一个优先级。当一个 tensor 被传递到 Sending Buffer 时，会被划分并存储为其优先级相对应的 FIFO 队列中的 Mercury 数据包。由此，提取具有最高优先级的数据包的时间复杂度为 O(N )，其中 N 是队列数。

在训练开始之前，worker 用户代码通知 Sending Buffer 来初始化 Mercury 数据包队列。文章发现大多数 DNN 模型的张量数量很少。因此，Mercucy 中为每个张量都分配了一个 Mercury 数据包队列。文章认为分配算法可以用来平衡调度多层 DNN 模型的效率和时间复杂度之间的权衡。

下图是 Mercucy 数据包的结构，包头指示其地址（张量 ID、偏移量）和优先级。每个包的有效载荷包含一个张量中的连续梯度或参数。不存在梯度或参数拆分成两个数据包，也不存在一个数据包含两个张量。为了避免内存复制，Sending Buffer 的 FIFO 队列中的每个 Mercury 数据包仅包含一个 header，而非实际数据。

Worker Sender 不断地从 Sending Buffer 中提取优先级最高的 Mercury 数据包（仅包头）。然后 Worker Sender 从原始 tensor 中找到数据，并使用传输库将 header 和数据传输到正确的 PS。

Transport Library 执行多路复用和多路分解操作，将 Mercury 数据包封装到多个传输库数据包（例如 TCP 数据包）中用于发送数据，并在接收数据时执行相反的过程。内存复制只发生在传输库中。Mercury 包的大小为 n × Spkt，其中 Spkt 是传输库包的大小（例如 TCP 最大段大小）。使用 n = 1 会增加调度系统的负担。因此，在 Mercucy 的实现中，n 设置为 10。

Mercucy PS

Transport Library：和 Mercucy Worker 里的一样

Packet Constructor：构造从 Worker 接收到的 Mercury 梯度数据包

Updater：聚合梯度并更新模型参数

PS Sender：将 Mercury 参数包传输给所有 Worker

Updater 是一个代理，负责保存 PS 的聚合逻辑和模型参数。 PS 用户代码在训练前将聚合函数和全局模型参数传递给 Updater。然后 PS 用户代码保持空闲，直到训练的完成信号被触发。所有的迭代模型同步工作都从用户代码转移到了 Updater。

Packet Constructor 构造 Mercury 梯度数据包，并在准备好时将其交付给 Updater。 Updater在收到所有 worker 的相同 header 的梯度包后，按照具体的聚合逻辑进行聚合。然后 Updater 更新一系列与梯度相对应的模型参数。

PS Sender 不断提取更新的参数，构造 M 个相同的 Mercury 参数包，其中 M 为 worker 个数。这些数据包分别传输给所有 Worker。

Packet Constructor、Updater 和 PS Sender 以流水线方式处理 Mercury 数据包。 Mercury 包的大小通常比张量小得多。因此 Mercury PS 能够近乎完美地实现上行链路和下行链路传输重叠。从 Worker 的角度来看，每一层的推拉都是完全重叠的，下文会展示此设计不影响 PS 的带宽利用率。

实例部署

文章基于内核 TCP 部署了 Mercury，并使用 C++ 和 Python 将其集成到 MXNet 框架中。

主要修改的是组件 Van，它是 ps-lite1 中点对点通信的抽象（即 MXNet 的底层 PS 架构库）。在当前的 ps-lite 实现中，push 和 pull 函数调用 Van 中的通信 API 来传输张量。文章使用 Mercury_Van 替换 Van，且没有对迭代训练的代码做任何修改，通信栈中的所有 API 都和以前一样。

文章在工作用户代码中添加了一个插件，能够在训练前通过环境变量通知 Mercury_Van 包队列的数量（即调度优先级的数量）。此外还在 PS 用户代码中添加了一个插件，用于初始化 Mercury_Van 中的聚合代理。

Mercury 能够很容易地与支持 PS 架构的其他深度学习框架集成，例如 Tensor-Flow 和 PyTorch。这些框架的 PS 架构实现维护所有节点的通信关系，并且通常建立在点对点通信库之上。为这些框架实现 Mercury，只需将它们的点对点通信库替换为 Mercury。 All-Reduce 架构同样可以从 Mercury 的基于优先级的调度中受益。
