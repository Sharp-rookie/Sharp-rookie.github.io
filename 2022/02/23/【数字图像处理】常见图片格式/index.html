【数字图像处理】常见图片格式

发表于 2022-02-23 分类于 技术分享 阅读次数： 阅读次数： Valine：

  图像格式即图像文件存放的格式，通常有JPEG、BMP、GIF、TIFF、RAW、PNG等。由于数码相机拍下的图像文件很大，储存容量却有限，因此图像通常都会经过压缩再储存

  最近比较流行的“10亿bit色深”屏幕、摄像头xxxx万像素等概念和图片存储格式相关，并且在机器学习、深度学习时数据预处理中，不同图片格式有不同的特点需要注意，因此这里总结一下常见图片格式的机理

 

JPEG格式

简介

  JPEG（联合图像专家组，Joint Photographic Experts Group）是JPEG标准的产物，正式名为：信息技术连续色调静止图像的数字压缩编码，该标准由国际标准化组织（ISO）制订，是面向连续色调静止图像的一种压缩标准。JPEG格式是最常用的图像文件格式，后缀名为 .jpg 或 .jpeg

  JPEG主要采用预测编码（DPCM）、离散余弦变换（DCT）以及熵编码的联合编码方式，以去除冗余的图像和彩色数据，属于有损压缩格式。它能够将图像压缩到很小的存储空间，压缩比例通常在 10:1 到 40:1 ，同一幅图，JPEG格式存储的文件是其他类型文件的不到十分之一，但这会在一定程度上造成图像数据的损伤。尤其是使用过高的压缩比例，将使最终解压缩后恢复的图像质量降低，如果追求高品质图像，则不宜采用过高的压缩比例。

JPEG格式分为3种：

  • 标准JPEG：

    在网页下载时只能由上而下依序显示图像，直到图像资料全部下载完毕，才能看到图像全貌

  • 渐进式JPEG：

    在网页下载时，先呈现出图像的粗略外观后，再慢慢地呈现出完整的内容；并且存成渐进式JPG格式的文档比存成标准JPG格式的文档要来得小，所以如果要在网页上使用图像，可以多用这种格式

  • JPEG2000：

    是新一代的影像压缩法，压缩品质更高，并可改善在无线传输时，常因信号不稳造成马赛克现象及位置错乱的情况，改善传输的品质

压缩标准

JPEG标准

  • 基于DPCM无损压缩

    采用预测法、哈夫曼编码，从而保证重建图像与原图像完全相同

  • 基于DCT的顺序编码

    根据DCT变换原理，按从上到下、从左到右的顺序对图像数据进行压缩编码。当信息传送到接收端时，首先按照上述规律进行解码，从而还原图像。在此过程中存在信息丢失，因此这是一种有损图像压缩编码

  • 基于DCT的累进编码

    也以DCT变换为基础的，但是其扫描过程不同。它通过多次扫描的方法来对一幅图像进行数据压缩。其描述过程采取由粗到细逐步累加的方式进行。图像还原时，在屏幕上首先看到的是图像的大致情况，而后逐步地细化，直到全部还原出来为止

  • 基于DCT的分层编码

    以图像分辨率为基准进行图像编码，首先是从低分辨率开始，逐步提高分辨率，直至与原图像的分辨率相同为止。图像重建时也是如此。可见其效果与基于DCT累进编码模式相似，但其处理起来更复杂，所获得的压缩比也更高一些

JPEG 2000标准

  JPEG采用离散余弦变换将图像压缩为8×8的小块，然后依次放入文件中，这种算法靠丢弃频率信息实现压缩，因而图像的压缩率越高，频率信息被丢弃的越多。在极端情况下，JFEG图像只保留了反映图貌的基本信息，精细的图像细节都损失了，为此，JPEG制定了新一代静止图像压缩标准 JPEG2000。

  JPEG 2000放弃了JPEG所采用的以离散余弦变换（DCT）为主的区块编码方式，而采用以小波变换为主的多解析编码方式，其主要目的是要将影像的频率成分抽取出来。小波转换将一幅图像作为一个整行变换和编码，很好地保存了图像信息中的相关性，达到了更好的压缩编码效果。

性能

当用质量与比特率之比来衡量时，JPEG的性能相当优越，其优点在于：

  • 支持极高的压缩率，JPEG图像的下载速度很快
  • 能够处理16.8M的颜色，可以很好地再现全彩的图像
  • 对图像压缩处理时，JPEG允许自由地在最小文件尺寸和最大文件尺寸之间选择
  • 文件尺寸相对较小，有利于在带宽并不富裕的情况下传输

JPEG的缺点：

  • 不是所有浏览器都支持各种JPEG图像插入网页
  • 压缩时，可能使图像的质量受到损失，故不适宜用于显示高清晰度图像

应用

  JPEG格式的压缩率是目前各种图像文件格式中最高的，压缩的主要是高频信息，对色彩的信息保留较好，适合应用于互联网；它可减少图像的传输时间，支持24位真彩色；也普遍应用于需要连续色调的图像中。

BMP格式

简介

BMP（全称Bitmap）是Window操作系统中的标准图像文件格式，能够被多种Windows应用程序所支持

BMP采用位映射存储格式，除了图像深度可选以外，不采用其他任何压缩，因此BMP文件所占用的空间很大

BMP文件的图像深度可选：1bit、4bit、8bit、24bit

BMP图像的扫描方式是按从左到右、从下到上的顺序

由于BMP文件格式是Windows环境中交换与图有关的数据的一种标准，因此在Windows环境中运行的图形图像软件都支持BMP图像格式

整体信息

BMP格式文件从头到尾依次是：

  • 位图文件头（bmp file header）：14 bytes
  • 位图信息头（bitmap information）：40 bytes
  • 调色板（color palette）：可选
  • 位图数据

最常见的就是24位图，即一个像素的颜色信息用24位来表示，对于三原色BRG，每一个颜色都用以字节（8）位来表示。除了24位图，还有1位（单色），2位（4色，CGA），4位（16色，VGA），8位（256色），16位（增强色），24位（真彩色）和32位等。

文件头

文件头包含文件格式、大小等信息

前14字节为bmp图像的文件头信息，其中：

字节 位名 含义 示例取值 说明
0~1 bfType 文件类型 4d 42 0x424d = “BM”，即 bmp 文件
2~5 bfSize 文件大小 00 04 a1 ae 0x4a1ae = 303534 bits
6~7 bfReserved1 保留位 00 00 固定值：0
8~9 bfReserved2 保留位 00 00 固定值：0
a~d bfOffBits 文件开头到图片数据的偏移量 00 00 00 36 0x36 = 54 bytes

（注意，数据存储采用小字节序，高位值存在高位地址）

信息头

信息头包含图像数据尺寸、位平面数、压缩方式、颜色索引等信息

第14~53字节为bmp图像的信息头分析，其中：

字节 位名 含义 示例取值 说明
0e~11 biSize 信息头大小 00 00 00 28 0x28 = 40 bits
12~15 biWidth 位图宽度 00 00 01 67 0x167 = 359 像素
16~19 biHeight 位图高度 00 00 01 19 0x119 = 281 像素
1a~1b biPlanes 平面数 00 01 0x1 = 1
1c~1d biBitCount 颜色深度位数 00 18 0x18 = 24 bit/像素
1e~21 biCompression 压缩类型 00 00 00 00 0—不压缩，即BI_RGB格式
22~25 biSizeImage 图像大小 = 宽×高×色深 00 00 00 00 (按理说不应该是359×281×24吗 ? )
26~29 biXPelsPerMeter 水平分辨率（像素/米） 00 00 0e c3 0xec2 = 3779 像素
2a~2d biYPelsPerMeter 垂直分辨率 00 00 0e c3 0xec2 = 3779 像素
2e~31 biClrUsed 调色板中颜色索引数 00 00 00 00 0—表示无调色板
32~35 biClrImportant 重要影响的颜色索引数 00 00 00 00 0—表示都重要

调色板

本例中，biBitCount=24、biClrUsed = 0，说明是24位真彩图，无调色板也无需调色板（4、8bit色深的才需要），位图信息头后面接的就是图像数据了；文件头14bytes、信息头40bytes，恰对应bfOddBites=54

如果色深不是24bit而是4bit、8bit，无论4bit还是8bit都不能反映R、G、B共8bits×3=24bits的颜色信息，因此必须使用调色板。调色板中的数据每4bytes一组代表一种颜色，分别表示蓝、绿、红、保留值Alpha（表示透明度，在24位RGB位图中恒定为0），例如：

而每个像素具体使用调色板中哪个颜色，取决于颜色位值（以颜色位值作为调色板的数组索引来选择颜色）

  • 4bit色深

    4 bits/像素，说明一个像素有16种可能的颜色值而调色板中一种颜色对应 一组4bytes 数据，因此8bit色深图像的调色板大小为 16×4bytes = 64bytes

  • 8bit色深

    8 bits/像素，说明一个像素有256种可能的颜色值，而调色板中一种颜色对应 一组4bytes 数据，因此8bit色深图像的调色板大小为 256×4bytes = 1024bytes

  • 16比特色深

    这种格式叫作高彩色，或叫增强型16位色，或64K色；它的情况比较复杂：

    当biCompression成员的值是BI_RGB时没有调色板，16位中，最低的5位表示蓝色分量，中间的5位表示绿色分量，高的5位表示红色分量，一共占用了15位，最高的一位保留为0

    当biCompression成员的值是BI_BITFIELDS时，情况就复杂了，原来调色板的位置被三个DWORD变量占据，称为红、绿、蓝掩码。分别用于描述红、绿、蓝分量在16位中所占的位置

图像数据

由于此图像是24位真彩图，没有调色板，所以信息头后直接就是图像数据

真彩图片每个像素都有24bits，即3bytes（R、G、B各1byte）；由于图片宽度为359像素，故每行有 359×3=1077bytes

但是要注意，Windows默认的扫描的最小单位是4字节，如果数据对齐满足这个值的话对于数据的获取速度等都是有很大的增益的。因此，BMP图像顺应了这个要求，要求每行的数据的长度必须是4的倍数，如果不够需要进行比特填充（以0填充），这样可以达到按行的快速存取。这样的话，位图数据的大小就不一定是宽x高x每像素字节数了，因为每行还可能有0填充

因此这张图片每行的大小应该是：1077 + 3 = 1080bytes（能被4整除）

GIF格式

简介

GIF格式的名称是Graphics Interchange Format的缩写，是在1987年由Compu Serve公司为了填补跨平台图像格式的空白而发展起来的，可以被 PC 和 Mactiontosh 等多种平台上被支持。

GIF采用Lempel-Zev-Welch（LZW）压缩算法，最高支持256种颜色，因此适用于色彩较少的图片，比如卡通造型、公司标志等。如果碰到需要用真彩色的场合，那么GIF的表现力就有限了。

GIF通常会自带一个调色板，里面存放需要用到的各种颜色。在Web运用中，图像的文件量的大小将会明显地影响到下载的速度，因此可以根据GIF带调色板的特性来优化调色板，减少图像使用的颜色数（有些图像用不到的颜色可以舍去）而不影响图片质量。

GIF格式和其他图像格式的最大区别在于，它完全是作为一种公用标准而设计的，由于Compu Serve网络的流行，许多平台都支持GIF格式。

整体信息

http://giflib.sourceforge.net/index.html（科学.上网）是一个比较完整详细的GIF格式参考资料，其中有一张介绍GIF文件内部组成的图：

其中共有11个模块，有8块是GIF图像必备的：

  • Header：头
  • Logical Screen Descriptor：逻辑屏幕描述符
  • Image Descriptor：图像描述符
  • Image Data：图像数据流
  • Plain Text Extension：文本拓展
  • Application Extension：应用拓展
  • Comment Extension：注释拓展
  • Trailer：尾部标记

另外3个是可选块：

  • Global Color Table：全局颜色表
  • Graphic Control Extension：图形控制拓展
  • Local Color Table：本地颜色表

传统GIF解码器正是通过上述模块来对GIF文件进行解析的

Logical Screen Descriptor

逻辑描述符通常跟在Header后面，告知解码器此GIF图像的分辨率、背景色和Global Color Table等信息，组成如下：

  • Canvas Width：GIF 图像的宽度（单位：像素）
  • Canvas Height：GIF 图像的高度（单位：像素）
  • Packed Field：包装字段，内部的不同 bit 各自含义如下
    1. Global Color Table Flag（1bit）：1—有、0—无
    2. Color Resolution（3bit）：颜色位深度，Global Color Table 中的 颜色数 = 2^(Color Resolution + 1)个，颜色位深度 = Color Resolution + 1
    3. Sort Flag（1bit）：1—Global Color Table 按照出现频率降序排列、0—不排序
    4. Size of Global Color Table：Global Color Table 的颜色数，等于 2^(Color Resolution + 1)
  • Background Color Index：GIF 背景色在 Global Color Table 中的索引
  • Pixel Aspect Retio：表示用于计算原始图像中像素宽高比的近似因子，一般情况为 0

上例中为：

Canvas Width = 0x01b0 = 432 像素

Canvas Height = 0x01b0 = 432 像素

Packed Field = 0xf6 = 11110110

  • Global Color Table Flag = 1，说明存在 Global Color Table
  • Color Resolution = 0b111 = 7，说明是 7+1 = 8 bit色深
  • Sort Flag = 0，说明未排序
  • Size of Global Color Table = 0b110 = 6，说明有2^(6+1) = 128

Background Color Index = 0，说明此 GIF 的背景色为 Global Color Table 中的第一个颜色

Pixel Aspect Ratio = 0，忽略

Graphic Control Extension

看见 Global Color Table 后面紧跟的那个字节，从示例中可以看到的21，21 在 GIF 格式中是有特殊意义的，它表示 Extension Introducer（扩展入口），即接下来的一段数据为某一种扩展

21 后的下一个字节是 f9，这也是一个标识符，代表这个拓展是 Graphics Control Extension

Graphics Control Extension 算上 21 和 F9 一共有8个字节，其结构如下图：

  • Byte Size：表示接下来的有效数据字节数（不带结束位）
  • Packed Field：包装字段，内部不同位的意义不同
    1. Reserved for Future Use（3bit）：保留位，暂无用处
    2. Display Method（3bit）：表示逐帧渲染时，前一帧留下的图像作何处理：
      • 0：不做任何处理
      • 1：保留前一帧图像，在此基础上进行渲染
      • 2：渲染前将图像置为背景色
      • 3：将被下一帧覆盖的图像重置
    3. User Input Flag（1bit）：表示是否需要在得到用户的输入时才进行下一帧的输入（具体用户输入指什么视应用而定）
      • 0——无需输入
      • 1——需要输入
    4. Transparent Color Flag（1bit）：1—— Transparent Color Index 指定的颜色当做透明色处理、0 ——不做处理
  • Delay Time：表示 GIF 动图每一帧之间的间隔，单位为百分之一秒；为 0 时，间隔由解码器管理
  • Transparent Color Index：为1时有效，此索引在 Global Color Table 中对应的颜色将当做透明色处理
  • Block Terminator：Extension 的结束位

上例中为：

  • 21 f9 表示这是一个 Graphic Control Extension 模块
  • Bytes Size = 4，有效长度为 4 bytes
  • Packed Field = 00000100
    1. Reserved for Future Use = 0
    2. Display Method = 1，表示逐帧渲染时，保留前一帧图像，在此基础上进行渲染
    3. User Input Flag = 0，下一帧的输入不需要得到用户的输入
    4. Transparent Color Flag = 0，Transparent Color Index 指定的颜色不做处理
  • Delay Time = 0x000a = 10ms
  • Transparent Color Index = 0xff = 255，无效，没有需要做透明处理的颜色值
  • Block Terminator = 00，结束位

Image Descriptor

Image Descriptor 的标识符（Image Seperator）是 2c，内部字节结构为：

  • Image Left：下一帧图像渲染位置离画布左边的距离（从 0 开始）
  • Image Top：下一帧图像渲染位置离画布上边的距离（从 0 开始）
  • Image Width：下一帧图像的宽度
  • Image Height：下一帧图像的高度
  • Packed Field：包装字段，内部不同位的意义不同
    1. Local Color Table Flag（1bit）：下一帧图像是否需要一个独立的颜色表，1—需要、0—不需要
    2. Interlace Flag（1bit）：是否需要隔行扫描，1—需要，0—不需要
    3. Sort Flag（1bit）：如果需要 Local Color Table 的话，这个字段表示其排列顺序
    4. Reserved For Future Use（2bit）：保留位，不用管
    5. Size of Local Color Table（3bit）：同 Global Color Table 中的该位。如需要本地颜色表，则该数有效

上例中为：

  • 2c 为标志位，表示这是一个 Image Descriptor 模块
  • Image Left = 0
  • Image Top = 0
  • Image Width = 0x01b0 = 432
  • Image Height = 0x01b0 = 432
  • Packed Field = 0b0000_0000

以上参数说明渲染的一帧大小为 432*432 像素，正好与 GIF 图的分辨率一致。打包字段都为零，表示下一帧不需要 Local Color Table，也不需要进行隔行扫描

Image Data

如果存在 Local Color Table，Image Data 就紧跟其后。如若不存在，则紧跟在 Image Descriptor 后。Image Data 的内部字节组成如下：

  • LZW Minimum Code Size：GIF 在对每一帧的像素颜色在 Color Table 所对应的索引进行 LZW 压缩，这里的 LZW Minimum Code Size 是 LZW 压缩中很关键的一个值
  • Number of bytes of data in sun-blocks：后面的有效字节的个数，范围为 01-FF。为 0则表示 Image Data 到此为止。由于其最大值为 FF，但图像像素数可能会大于这个值，所以从图上也能知道这个 Data sub-Blocks有可能接连出现很多个
  • Data Sub-Block(s)：有效的字节块
  • Block Terminator：结束符
